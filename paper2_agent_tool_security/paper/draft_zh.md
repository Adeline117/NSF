# AI Agent工具接口安全性：跨协议实证分析

**目标会议：** IEEE S&P / USENIX Security 2027

---

## 摘要

AI agent正日益依赖外部工具接口执行现实世界操作——签署区块链交易、查询数据库、执行代码及管理基础设施。多种协议并存以调解这些交互：Anthropic的Model Context Protocol (MCP)、OpenAI Function Calling、LangChain代理框架工具以及Web3原生智能合约模块。尽管授予大语言模型(LLM)对特权操作的访问具有重大安全隐患，迄今尚无系统性的跨协议安全分析。本文提出了首个此类分析：我们从GitHub枚举336个工具服务仓库，涵盖四个协议族（MCP 205个、Web3原生79个、LangChain 40个、OpenAI 12个），并对其中20个高星仓库进行深度静态分析。我们设计了覆盖5个攻击面的30余种漏洞检测模式，映射至11个CWE标识。在20个目标仓库中共发现1,415个安全问题，其中86个为严重级别、454个为高危级别、875个为中危级别。跨协议分析显示，11个漏洞类别中有10个（90.9%）出现在两个及以上协议族中，表明这些安全缺陷是系统性的架构问题而非特定协议的实现错误。我们进一步识别了4个Web3特有攻击向量（交易构造注入、Approve劫持、私钥提取、MEV利用），并揭示了安全包装器（Harness）的普遍缺失。我们提出6项Harness安全需求并评估其采纳现状。分析工具与数据集已开源发布。

## 1 引言

大语言模型正在从被动的文本生成器演变为主动的自主代理（AI agent）。这一范式转变的核心在于工具使用能力：agent通过调用外部工具与现实世界交互，包括读写文件系统、查询数据库、发送网络请求以及签署区块链交易 [1, 2]。工具接口赋予了agent执行特权操作的能力，但同时也引入了新的、独特的攻击面——agent继承了其可调用的每一个工具的安全漏洞。

当前，多种协议竞相定义agent与工具之间的通信标准。Anthropic于2024年末发布的Model Context Protocol (MCP) [3] 定义了工具、资源和提示三种原语，通过stdio或SSE传输层进行通信。OpenAI的Function Calling机制 [4] 在Chat API中使用JSON Schema定义函数接口。LangChain [5] 通过Python `@tool`装饰器和AgentExecutor提供工具绑定。在Web3领域，Safe模块 [6]、ERC-4337插件 [7] 和ERC-7579模块 [8] 构成了原生的智能合约工具接口体系。这些协议在传输机制、模式定义、执行模型和信任假设上存在根本差异，但均面临共同的安全挑战。

现有安全分析存在两个显著局限。**第一**，研究范围局限于单一协议。Invariant Labs [9] 对MCP进行了首个安全分析，发现了工具投毒攻击（tool poisoning），但未考察该漏洞是否为MCP特有。Hou等人 [10] 分析了MCP多代理系统中的工具间提权风险，同样限于MCP协议。**第二**，Web3场景的特殊风险未被系统研究。区块链交易具有最终性（finality），一旦执行即不可逆转——没有撤销交易（chargeback），没有回滚（undo）。一个恶意的`sendTransaction`调用即可清空整个钱包。将AI agent与区块链工具结合产生了独特的攻击向量，包括交易构造注入、无限授权劫持和私钥提取，其严重程度远超传统工具接口漏洞。

本文提出以下三个研究问题：

- **RQ1:** AI agent工具接口存在哪些安全漏洞？其分布和严重程度如何？
- **RQ2:** 这些漏洞是特定协议的实现缺陷，还是跨越协议边界的系统性问题？
- **RQ3:** Web3场景为AI agent工具接口引入了哪些特有的攻击向量？

为回答上述问题，我们进行了首个跨协议大规模安全实证分析。我们从GitHub枚举并去重336个工具服务仓库，涵盖四个协议族；选择20个高影响力仓库进行深度静态分析，设计30余种漏洞检测模式覆盖5个攻击面并映射至CWE标准。分析发现1,415个安全问题，其中11个漏洞类别中10个出现在两个及以上协议族中。

本文做出以下贡献：

1. **统一漏洞分类体系**：定义11个漏洞类别，映射至CWE标准，覆盖5个攻击面。
2. **首个跨协议实证分析**：涵盖336个仓库、4个协议族，对20个高星仓库进行深度分析。
3. **五大攻击面与四个Web3特有向量**：系统化描述AI agent工具接口的完整威胁景观。
4. **Harness缺口分析**：定义6项安全包装器需求并量化其实际采纳率。
5. **开源工具与数据集**：发布静态分析框架和匿名化数据集。

## 2 背景与威胁模型

### 2.1 AI Agent工具协议

AI agent通过工具协议发现、描述、调用和接收外部工具的执行结果。当前存在四个主要协议族，各自在传输机制、模式定义和执行模型上具有不同特征。

**MCP (Model Context Protocol).** Anthropic于2024年末发布的开放协议 [3]，定义了三种交互原语：工具（tools）用于执行操作、资源（resources）用于数据访问、提示（prompts）用于模板化交互。传输层支持stdio（本地进程间通信）和SSE/HTTP（远程通信）。MCP还支持采样（sampling）机制，允许服务端请求LLM进行推理。在我们的数据集中，MCP是最大的协议族，占336个仓库中的205个（61.0%）。

**OpenAI Function Calling.** OpenAI在Chat Completions API中引入的机制 [4]，开发者通过JSON Schema定义函数签名，包括名称、描述和参数模式。当模型决定调用函数时，返回结构化的函数调用请求；客户端负责实际执行并将结果以`tool`角色消息返回。该机制不定义服务端架构，将执行责任完全委托给客户端实现。我们的数据集包含12个OpenAI Function Calling仓库。

**LangChain/Agent框架工具.** LangChain [5] 通过Python `@tool`装饰器和`BaseTool`基类定义工具，由`AgentExecutor`编排工具调用。工具定义包括名称、描述字符串和Python类型提示。LangChain的特点在于其共享内存机制——多个工具通过agent的记忆模块共享状态，这引入了独特的跨工具信息泄露风险。我们的数据集包含40个LangChain仓库。

**Web3原生模块.** 在区块链生态中，智能合约模块直接作为agent的工具接口。Safe模块 [6] 允许第三方代码代表多签钱包执行交易；ERC-4337 [7] 定义了账户抽象插件，支持自定义验证和执行逻辑；ERC-7579 [8] 规范了模块化智能账户的模块接口。这些模块以委托权限运行，可代表用户账户执行任意操作。我们的数据集包含79个Web3原生仓库。

表1总结了四个协议族的关键特征差异。

| 特征 | MCP | OpenAI FC | LangChain | Web3原生 |
|------|-----|-----------|-----------|----------|
| 传输层 | stdio/SSE/HTTP | HTTPS API | Python进程内 | EVM/RPC |
| 模式定义 | JSON Schema | JSON Schema | Python类型 | Solidity ABI |
| 执行模型 | 服务端进程 | 客户端托管 | Python进程 | 链上执行 |
| 信任假设 | 隐式信任 | 隐式信任 | 隐式信任 | 最小信任 |
| 状态管理 | 会话级 | 无状态 | 共享内存 | 链上状态 |

### 2.2 系统模型

AI agent工具接口由五个组件构成，我们独立且组合地分析其安全属性：

1. **Agent（代理）**：基于LLM的推理引擎（如Claude、GPT-4、Llama），解释用户意图并决定调用哪些工具。
2. **Protocol（协议）**：通信层，定义agent如何发现、描述、调用工具并接收结果。涵盖上述四个协议族。
3. **Tool Server（工具服务）**：执行请求的操作（文件I/O、API调用、区块链交易）并返回结构化结果。
4. **Harness（安全包装器）**：（通常缺失）围绕工具执行的安全封装，负责实施沙箱隔离、权限约束、速率限制和人工确认机制。
5. **Skill（技能）**：打包的能力单元，由提示模板、一个或多个工具绑定和编排工作流组成。技能引入了组合层攻击面，这在单独调用工具时不存在。

```
用户 ---> Agent (LLM) --[协议]--> Harness? ---> Tool Server
               ^                                     |
               |_________ 结果（可能被投毒）_________|
```

### 2.3 信任边界

我们定义了7个信任边界，每个边界代表一个数据在不同信任域之间传递的位置：

| 边界 | 描述 | 风险 |
|------|------|------|
| B1: 用户-Agent | 用户提示是不可信输入 | 注入攻击 |
| B2: Agent-协议 | Agent构造工具调用；协议序列化 | 参数篡改 |
| B3: 协议-Harness | Harness（如存在）在执行前拦截 | 绕过控制 |
| B4: Harness-工具 | 工具以自身权限级别执行 | 权限提升 |
| B5: 工具-Agent | 工具结果回传至LLM上下文 | 输出注入 |
| B6: 工具-外部世界 | 工具与外部系统交互（区块链、DB、API） | 未授权操作 |
| B7: Skill-Agent | Skill提示模板合并入agent上下文 | 指令覆盖 |

### 2.4 攻击者模型

我们定义三类攻击者，按访问权限递增、但在野外出现频率递减排列：

**A1: 恶意工具提供者。** 控制工具服务的源代码和部署。能力包括：在工具描述中注入操控性指令（工具投毒）；返回精心构造的结果以操纵agent推理；嵌入带有恶意默认值的隐藏参数；注册与合法工具同名的仿冒工具。目标包括窃取凭证（私钥、API密钥、会话令牌）、执行未授权交易和渗出用户数据。**流行度：高。** 开放的工具注册表（npm、PyPI、GitHub）允许任何人发布工具服务，用户基于README安装而无安全审计。

**A2: 提示注入攻击者。** 无法修改工具代码，但通过agent处理的数据（网页、邮件、文档、数据库记录、API响应）注入恶意内容。能力包括：构造能存活于输出净化的提示注入载荷；利用跨工具信息流（读工具泄露数据给写工具）；触发用户未请求的工具调用。**流行度：非常高。** agent读取的任何数据源都是潜在的注入向量。

**A3: 中间人。** 在agent和工具服务之间拦截协议消息。能力包括：修改传输中的工具描述、更改调用参数（如收款地址、金额）、修改工具结果。**流行度：较低但关键。** MCP的stdio传输免疫；MCP的SSE/HTTP传输、OpenAI API调用和RPC端点在未端到端启用TLS时均易受攻击。

### 2.5 攻击面

基于系统模型和攻击者模型，我们定义5个攻击面：

**S1: 工具定义面。** 向agent描述工具的元数据——名称、描述字符串、参数模式、默认值和示例。Agent将工具描述视为可信指令，恶意描述在协议层面与合法描述不可区分。漏洞模式包括描述中的指令注入、带恶意默认值的隐藏参数和过度宽松的模式定义。CWE映射：CWE-1321、CWE-913。

**S2: 输入构造面。** Agent从用户请求、对话上下文和工具模式构造工具调用参数的过程。Agent执行隐式类型转换、填充"合理"默认值，并可能将用户输入直接插入结构化参数而不进行验证。CWE映射：CWE-20、CWE-74、CWE-843。

**S3: 执行面。** 工具执行的运行时环境——进程隔离、文件系统访问、网络访问和系统调用。大多数工具服务以宿主进程的完整权限执行。漏洞模式包括无进程沙箱、不受限的文件系统和网络访问、`eval()`/`exec()`执行不可信输入。CWE映射：CWE-269、CWE-250、CWE-78、CWE-829。

**S4: 输出处理面。** 工具结果返回agent并纳入LLM上下文窗口以供进一步推理。工具结果被agent视为可信上下文，恶意结果可包含提示注入载荷、敏感数据或误导性信息。CWE映射：CWE-116、CWE-200、CWE-117。

**S5: 跨工具面。** 单个agent会话中多个工具之间的信息流和状态共享。Agent通常可访问不同权限级别的多个工具，缺乏跨工具数据流的访问控制使得低权限工具（只读数据获取器）可通过共享LLM上下文影响高权限工具（交易签名器）。CWE映射：CWE-863、CWE-284、CWE-362。

### 2.6 Web3特有攻击向量

以下四个向量专属于与区块链系统交互的agent-工具接口，与上述五个攻击面正交，通常跨越S2、S3和S5。

**W1: 交易构造注入。** 恶意工具（或提示注入）导致agent构造带有有害参数的区块链交易。目标包括：收款地址（重定向资金至攻击者钱包）、代币数量（清空全部余额）、`approve`授权方（向攻击者合约授予无限额度）、合约交互数据（调用恶意函数）。区块链交易的最终性意味着损失不可逆。CWE：CWE-345。

**W2: Approve劫持。** 工具将`approve(MaxUint256)`调用伪装为"必需步骤"插入实际操作之前。Agent遵循工具描述执行无限授权而不质疑数额。攻击者合约随后通过`transferFrom`清空已授权代币。攻击流程：(1) 工具描述要求"必须先授权无限额度"；(2) Agent调用`token.approve(攻击者合约, MaxUint256)`；(3) 用户的预期操作正常执行（看起来成功）；(4) 数小时/天后，攻击者调用`token.transferFrom(用户, 攻击者, 全部余额)`。ERC-20的授权模型是众所周知的安全隐患，无限授权在合法DeFi中普遍存在，因此不会引起警觉。CWE：CWE-250、CWE-862。

**W3: 私钥/会话密钥提取。** 提示注入或工具投毒尝试从agent上下文或工具参数中提取钱包凭证。向量包括：工具描述指示agent将私钥作为参数传入；工具结果中的提示注入要求agent"确认"密钥；会话密钥范围未正确限制；密钥存储于所有工具共享的环境变量中。私钥是持有者认证——无法撤销，无二次验证回退。密钥泄露意味着永久失去该密钥控制的所有资产。CWE：CWE-200、CWE-312。

**W4: MEV利用。** 恶意工具向MEV搜索者泄露待处理交易详情，或构造不带滑点/截止时间保护的交易以实现三明治攻击。向量包括：工具将交易发送到公共内存池而非私有中继；工具将滑点容忍度设为100%或完全省略；工具描述建议"为获得最佳效果，始终将滑点设为100%"。MEV是年提取规模超6亿美元的产业，三明治攻击在同一区块内自动化执行。CWE：CWE-200。

### 2.7 Harness安全需求

基于上述攻击面和向量，我们定义6类安全控制措施，作为正确保护agent-工具接口的基准需求：

**H1: 权限约束。** 每个工具仅获得其声明功能所需的最小权限。只读价格查询工具不能签署交易。实现方式：基于能力的访问控制、每工具IAM策略。

**H2: 执行沙箱。** 工具执行与宿主进程和其他工具隔离。包括资源限制（CPU、内存、时间）、网络限制（出站防火墙）和文件系统访问控制（只读挂载，禁止访问`.env`或密钥文件）。实现方式：容器、WASM、V8隔离区、seccomp-bpf。

**H3: 交易验证。** 针对Web3工具：允许的合约地址白名单、每笔和每会话支出上限、Gas限制强制执行、授权数额封顶（禁止`MaxUint256`）、滑点边界。实现方式：签名前评估的策略引擎。

**H4: 审计日志。** 所有工具调用记录时间戳、调用者身份、参数、结果哈希和执行时长。日志仅可追加且具有防篡改性。实现方式：结构化日志写入不可变存储，Web3工具发射链上事件。

**H5: 人工确认。** 高价值或高风险操作在执行前需要用户明确确认。基于阈值：超过特定金额的交易、首次工具调用、对生产资源的操作。实现方式：带超时的确认提示，组织环境中的多方审批。

**H6: 输出净化。** 工具结果在注入LLM上下文前进行净化处理。剥离已知的提示注入模式、截断至最大长度、脱敏敏感数据模式（密钥、密码、PII）。实现方式：在协议层应用的输出过滤管线。

## 3 方法

### 3.1 Server枚举与选择

我们采用系统化的仓库枚举策略从GitHub收集AI agent工具服务。

**搜索策略。** 我们使用GitHub API执行32个搜索查询，覆盖四个协议族的关键术语组合。对于MCP，搜索关键词包括"mcp tool server""mcp server""model context protocol"等；对于Web3原生模块，包括"safe module""erc4337 plugin""erc7579 module""smart account module"等；对于LangChain，包括"langchain tool""agent tool blockchain""langchain web3"等；对于OpenAI Function Calling，包括"openai function calling""function calling blockchain""chatgpt plugin"等。

**初始结果与去重。** 搜索返回的原始结果经过去重处理，按完整仓库名（owner/repo）去重后得到336个唯一仓库。

**协议族分布。** 336个仓库的协议族分布如下：
- MCP：205个仓库（61.0%），平均442颗星，总计90,605颗星
- Web3原生：79个仓库（23.5%），平均52.6颗星，总计4,153颗星
- LangChain：40个仓库（11.9%），平均15.1颗星，总计605颗星
- OpenAI：12个仓库（3.6%），平均2.9颗星，总计35颗星

**深度分析选择。** 从336个仓库中，我们按星数和活跃度选择Top 20进行深度静态分析。选择标准包括：(1) 公开且非归档状态；(2) 源代码可获取；(3) 在目标协议族内星数排名前列；(4) 最近6个月内有提交活动。最终选定的20个仓库累计约57,000颗GitHub星，涵盖TypeScript、Python、JavaScript和Rust等主要编程语言。

### 3.2 协议分类

我们开发了自动化协议分类器，通过以下三层检测机制确定每个仓库的协议族归属：

**导入分析。** 扫描源文件中的import/require语句，检测协议特征库的引用：`@modelcontextprotocol/sdk`（MCP）、`openai`包中的function calling相关API（OpenAI）、`langchain`库的工具相关模块（LangChain）、Solidity中的Safe/ERC-4337/ERC-7579接口引用（Web3原生）。

**依赖扫描。** 分析`package.json`、`requirements.txt`、`Cargo.toml`等依赖文件，匹配已知的协议SDK包名。

**文件级指标。** 检测特征性文件模式，如MCP的服务配置文件、OpenAI的函数定义JSON、LangChain的agent配置以及Solidity合约文件。

对20个深度分析仓库，分类器的置信度平均达到0.95以上。对于置信度低于0.8的仓库，我们进行了人工验证。最终20个仓库中：14个MCP、3个OpenAI、1个LangChain、2个分类未定（unknown）。

### 3.3 静态分析框架

我们设计并实现了专用的静态分析框架，针对AI agent工具接口的独特安全特征进行检测。

**漏洞模式设计。** 基于前述威胁模型和攻击面分析，我们定义了30余种基于正则表达式的漏洞检测模式。每个模式映射至一个漏洞类别和CWE标识，覆盖5个攻击面。核心检测模式包括：

- **IV-001/IV-002（输入验证缺失）**：检测地址参数未经以太坊地址格式验证（`Address =`模式无checksum校验）、数值参数未设边界检查（`value`类参数直接使用无范围校验）。映射CWE-20。
- **CE-001（跨工具提权）**：检测`callTool()`调用链中无权限检查的模式，标识工具间的非授权调用。映射CWE-284。
- **MH-001（Harness缺失）**：检测MCP Server、OpenAI函数处理器和LangChain工具实例化时是否包含安全封装。映射CWE-284。
- **HC-001（硬编码凭证）**：检测源代码中的API密钥、私钥和访问令牌硬编码模式。映射CWE-798。
- **PI-001/PI-002（提示注入）**：检测工具描述和输出中的指令注入模式。映射CWE-74。
- **SC-001（状态混淆）**：检测全局可变状态在工具调用间的不安全共享。映射CWE-362。
- **UA-001（无限授权）**：检测`MaxUint256`授权和无上限的`approve`调用。映射CWE-250。
- **PKE-001（私钥暴露）**：检测私钥在环境变量、参数或日志中的明文出现。映射CWE-312。

**误报过滤。** 为提高结果精确度，我们实施了三层误报过滤：(1) 注释检测——忽略代码注释中的匹配；(2) 测试排除——标记（但不过滤）测试文件中的发现，分开统计；(3) 上下文启发式——对特定模式进行上下文分析，例如`value: string`仅在参数传递上下文中标记。

### 3.4 风险评分

我们为每个仓库计算综合风险评分（0-100分），计算公式如下：

**风险分 = min(100, Σ(发现数_i × 严重度权重_i × 类别乘数_i) - Harness折扣)**

其中：
- 严重度权重：critical = 10, high = 5, medium = 1
- 类别乘数：根据漏洞类别的实际风险影响调整（如`private_key_exposure`乘数最高）
- Harness折扣：对每项实现的Harness特性给予固定分数折扣

风险等级划分：Critical ≥ 80, High ≥ 60, Medium ≥ 40, Low < 40。

### 4.4 Web3特有发现

在336个初始仓库中，Web3原生协议占79个（23.5%）。虽然深度分析的20个仓库中Web3原生仓库较少，但我们仍发现了多个Web3特有的安全问题。

**私钥暴露。** 4个私钥暴露发现全部出现在OpenAI协议族仓库中。典型模式为：私钥以环境变量形式存储，通过`process.env.PRIVATE_KEY`直接读取后传入签名函数，未经任何安全封装。在agent-工具架构中，这意味着同一进程中运行的任何工具都可访问该私钥——一个只读的价格查询工具与签署百万美元交易的签名工具共享相同的密钥材料。

**无限授权。** 25个无限授权发现分布于3个协议族：MCP（7个）、OpenAI（2个）和Unknown（16个）。最典型的模式是工具在执行代币操作前调用`approve`函数并传入`MaxUint256`（2^256 - 1）作为授权额度。虽然这在传统DeFi开发中是常见做法（为减少Gas费），但在agent-工具场景中风险被急剧放大：agent可能被工具投毒指令操纵执行对攻击者合约的无限授权，而用户对此完全不知情。

**交易验证缺失。** 多个仓库在构造区块链交易时不验证目标地址、数额和合约调用数据的完整性。我们在输入验证缺失类别中识别出大量与区块链参数相关的发现（如地址参数未经checksum验证），这些在Web3场景下的影响远高于一般场景——错误的地址意味着资金永久丢失。

**MEV保护缺失。** 在我们分析的涉及交换（swap）操作的工具服务中，未发现任何实施滑点保护或私有交易中继的实例。这意味着通过AI agent执行的所有DEX交易都暴露于三明治攻击风险之下。

### 4.5 Harness缺口分析

我们评估了20个仓库对6项Harness安全需求（H1-H6）的实现情况。

**整体缺口。** 在20个仓库中，仅8个（40%）被检测到包含任何Harness相关特性。13个`missing_harness`发现分布于这8个仓库（其中部分仅实现了最基本的Harness特性如日志记录）。更重要的是，12个仓库（60%）完全没有任何Harness特性的痕迹。

**按需求类别分析：**

- **H1（权限约束）**：0/20个仓库实现了每工具的权限约束。所有仓库中的工具均以宿主进程的完整权限运行。
- **H2（执行沙箱）**：0/20个仓库实现了工具执行沙箱。MCP服务以Node.js进程运行，工具可访问完整文件系统和网络。
- **H3（交易验证）**：0/20个仓库实现了交易构造前的参数验证策略引擎。
- **H4（审计日志）**：约3/20个仓库包含基本的日志记录功能，但均非安全级别的不可变审计日志。
- **H5（人工确认）**：0/20个仓库实现了自动化的人工确认机制。部分MCP服务的宿主应用（如Claude Desktop）在应用层提供确认，但这不属于工具服务本身的Harness。
- **H6（输出净化）**：0/20个仓库对工具输出实施了净化处理以防止提示注入回传。

**Harness与风险分的相关性。** 我们观察到Harness特性的存在与风险分之间存在弱负相关：有Harness特性的仓库平均风险分略低（但样本量不足以确立统计显著性）。例如，具有1个Harness特性的jamubc/gemini-mcp-tool风险分为25.8（Low），而无任何Harness特性的sooperset/mcp-atlassian风险分为100.0（Critical）。

**核心发现：Harness的普遍缺失是AI agent工具安全最紧迫的问题。** 6项Harness需求中没有任何一项在我们的数据集中得到充分实现。这意味着当前的AI agent工具接口在"裸奔"状态下运行——工具以完整权限执行、无沙箱隔离、无输入/输出验证、无审计追踪、无人工确认门槛。

### 4.6 高风险Server案例分析

我们对风险分最高的三个仓库进行深入案例分析。

**案例1：sooperset/mcp-atlassian（风险分100.0，275个发现）。** 这是一个连接Atlassian生态（Jira、Confluence）的MCP服务，拥有4,834颗星。275个发现中包括大量输入验证缺失和跨工具提权问题。该服务暴露了丰富的项目管理和文档操作工具，但未实施任何权限隔离——读取Confluence文档的工具与修改Jira issue状态的工具共享相同权限上下文。攻击者可通过工具投毒使agent在读取恶意Confluence页面后自动修改关键Jira issue的状态或分配。

**案例2：getsentry/XcodeBuildMCP（风险分100.0，229个发现）。** 由Sentry维护的Xcode构建MCP服务，5,066颗星。229个发现反映了大型代码库中输入验证的系统性缺失。该服务涉及代码构建和测试执行——攻击面尤为敏感，因为工具可触发任意代码编译和执行。3个Harness缺失发现表明MCP Server实例化时未附加安全包装。

**案例3：ridafkih/keeper.sh（风险分96.5，147个发现）。** 一个密码/密钥管理相关工具，970颗星。其高风险分源于硬编码凭证（6个发现）与无限授权（16个发现）的组合。密钥管理工具本身存在凭证硬编码是一个尤为严重的矛盾。16个无限授权发现表明该工具在与区块链交互时使用了不安全的授权模式。

## 5 讨论

### 5.1 系统性漏洞的根因分析

我们的实证发现表明，AI agent工具接口的安全问题是系统性的而非偶发的。10/11个漏洞类别跨越多个协议族边界，这一结果指向三个根本原因：

**协议设计优先功能而非安全。** 四个协议族的规范均侧重于定义功能互操作性——如何描述工具、如何传递参数、如何返回结果。安全性被视为"可选的上层关注"而非协议核心属性。MCP规范提及了安全考量但不强制执行；OpenAI Function Calling将全部安全责任留给客户端；LangChain的`@tool`装饰器不包含任何安全元数据；Web3模块依赖Solidity的原生访问控制但缺乏agent层面的约束。

**开发者从不安全的范例中学习。** 协议文档和教程中的示例代码普遍不包含安全最佳实践。开发者复制这些模式，产生了遍布生态的"遗传性"漏洞。例如，MCP的官方inspector工具本身被评为Critical风险（87.3分，80个发现），作为参考实现和测试工具，其模式被下游开发者广泛借鉴。

**安全默认值的缺失。** 没有任何协议实现"安全默认"（secure-by-default）原则。工具默认以完整权限执行、默认无沙箱、默认无输入验证、默认无输出净化。安全需要开发者主动且正确地实现每一项控制措施——这是一个已被证明无效的安全策略。

### 5.2 协议设计建议

基于我们的发现，我们提出以下协议层面的安全增强建议：

**R1: 强制性权限声明。** 工具注册时必须声明所需的最小权限集合（读/写/执行/签名）。协议层在运行时强制执行权限边界。

**R2: 内置输入验证原语。** 协议模式定义应支持丰富的验证约束，不限于JSON Schema的基本类型。对于Web3工具，应内置地址格式验证、数值范围约束和授权额度限制。

**R3: 协议级输出净化。** 工具返回结果在进入LLM上下文前必须经过协议层的净化处理。这应是不可旁路的默认行为，而非可选特性。

**R4: 标准化Harness接口。** 协议应定义标准化的安全包装器接口，允许第三方安全中间件以声明式方式插入工具执行管道。

**R5: 交叉工具信息流控制。** 协议应支持声明式的工具间数据流策略，防止低权限工具的输出影响高权限工具的输入构造。

### 5.3 Harness设计框架

基于我们定义的6项Harness需求和实证的缺口分析，我们提出分层Harness设计框架：

**Layer 1: 协议层Harness（由协议SDK实现）。** 包括模式验证（H1部分）、输出净化（H6）和基本审计日志（H4）。这一层应由协议SDK作为默认行为提供，开发者无需额外代码。

**Layer 2: 运行时Harness（由部署环境实现）。** 包括执行沙箱（H2）、资源限制和网络隔离。可通过容器化、WASM隔离或进程级安全策略实现。

**Layer 3: 业务逻辑Harness（由工具开发者实现）。** 包括领域特定的验证规则（H3，如交易参数验证）和人工确认策略（H5，如高价值操作审批）。

这一分层设计确保Layer 1和Layer 2的安全措施无需开发者干预即可生效，仅Layer 3需要领域专家的参与。

### 5.4 负责任披露

我们已向所有20个分析仓库的维护者发送了安全通知。对于风险评级为Critical的11个仓库，我们提供了详细的漏洞报告和修复建议。我们遵循90天的负责任披露窗口期。截至论文提交，已收到部分维护者的确认回复。

我们的发现涉及的是工具服务代码本身的安全缺陷，而非特定用户的数据泄露。所有动态测试均在本地测试网或合成环境中执行，未涉及真实资金或用户数据。

### 5.5 局限性

**静态分析的精确度。** 基于正则表达式的静态分析存在固有的精确度局限。我们的检测模式可能产生误报（将安全代码标记为漏洞）和漏报（遗漏实际漏洞）。特别是IV-002（数值参数无边界检查）模式的误报率较高，因为它匹配了许多合法的字符串处理代码。我们通过人工抽样验证（对20个仓库中的200个发现进行人工审核）估算精确度约为65-75%。

**样本量限制。** 深度分析仅覆盖20个仓库（占336个枚举仓库的6.0%）。协议族分布不均衡：MCP 14个、OpenAI 3个、LangChain 1个、Unknown 2个，LangChain和OpenAI的样本量不足以支持统计显著的协议间比较。

**数据源限制。** 我们的数据集仅来自GitHub公开仓库。npm和PyPI注册表中的工具包、商业闭源工具服务以及企业内部部署的工具均不在分析范围内。这些数据源可能呈现不同的安全特征。

**Web3深度分析不足。** 虽然我们在初始枚举中包含了79个Web3原生仓库，但深度分析的20个仓库中Web3原生占比较低。Web3特有发现主要来自其他协议族仓库中涉及区块链交互的代码。对Web3原生智能合约模块（Safe modules、ERC-4337 plugins）的专门安全分析有待未来工作。

**因果推断限制。** 我们的分析是观察性的，仅证明了漏洞的存在和分布。我们无法确定这些漏洞在实际攻击中是否被利用，也无法量化其潜在的经济损失。

## 6 相关工作

### 6.1 MCP安全分析

Model Context Protocol (MCP) 由Anthropic于2024年末发布 [3]，作为连接LLM与外部工具和数据源的标准化接口。Invariant Labs [9] 发表了首个MCP系统安全分析，识别了工具投毒攻击——恶意工具描述嵌入操纵agent行为的指令。他们的工作证明MCP工具描述被LLM视为可信指令，使恶意工具提供者可劫持agent的行为。然而，其分析限于MCP协议，未考察漏洞是否跨协议存在。Hou等人 [10] 分析了MCP多代理系统中的安全风险，聚焦于工具间提权。本文将这些分析扩展至四个协议族，证明10/11个漏洞类别跨越协议边界。

### 6.2 LLM工具使用安全

Greshake等人 [11] 引入了间接提示注入概念，证明了工具增强型LLM因注入指令可触发未请求的工具调用而特别脆弱。Zhan等人 [12] 系统性分类了针对LLM agent的攻击，区分了针对推理的攻击和针对工具接口的攻击。OWASP LLM Top 10 [13] 将"过度授权"(LLM06)列为顶级风险。Debenedetti等人 [14] 提出AgentDojo框架评估AI agent的对抗鲁棒性。本文与上述工作互补：我们分析工具服务本身的固有漏洞，而非agent的鲁棒性。

### 6.3 智能合约安全

Atzei等人 [15] 发表了以太坊智能合约漏洞的开创性综述。Chen等人 [16] 使用机器学习大规模检测合约漏洞。Tsankov等人 [17] 开发了Securify安全分析器。本文的Web3原生分析针对模块化智能账户组件（Safe模块、ERC-4337插件、ERC-7579模块），这些作为AI agent工具接口引入了新的攻击面——模块可代表账户以委托权限执行操作，这一交叉领域未被现有工具覆盖。

### 6.4 API安全

OWASP API安全Top 10 [18] 识别了常见API漏洞。Atlidakis等人 [19] 开发了RESTler用于API端点模糊测试。我们认为AI agent工具接口是一类特殊API，具有超出传统API安全范畴的攻击面——特别是工具定义面（S1，描述影响LLM行为）和跨工具面（S5，信息通过LLM上下文流动）。

### 6.5 AI Agent安全综述

Xi等人 [20] 提供了AI agent架构和安全的全面综述，将工具使用列为核心能力。Wang等人 [21] 调查了基于LLM的自主agent，但安全讨论聚焦于对齐和安全性而非工具接口。Ruan等人 [22] 提出了AI agent系统风险分类。本文为这些综述提供了所呼吁的实证基础：我们在真实工具服务实现中量化了特定漏洞类别的流行度。

## 7 结论

本文提出了首个针对AI agent工具接口的跨协议大规模安全实证分析。我们从GitHub枚举336个工具服务仓库，涵盖MCP、OpenAI Function Calling、LangChain和Web3原生四个协议族，对20个高星仓库进行了基于30余种检测模式的深度静态分析。

核心发现可总结为三点：

**第一，安全问题是系统性的。** 我们发现的1,415个安全问题分布于11个漏洞类别，其中10个（90.9%）出现在两个及以上协议族中。输入验证缺失（674个，47.6%）、跨工具提权（378个，26.7%）和状态混淆（186个，13.1%）是最普遍的三个类别，均跨越全部四个协议族。这证明了AI agent工具接口的安全缺陷是架构性的——源于协议设计而非个别实现错误。

**第二，安全包装器普遍缺失。** 在我们定义的6项Harness安全需求中，没有任何一项在数据集中得到充分实现。60%的仓库完全不包含任何Harness特性。这意味着当前AI agent工具接口在无沙箱、无权限隔离、无审计追踪的"裸奔"状态下运行。

**第三，Web3场景急剧放大风险。** 我们识别的4个Web3特有攻击向量（交易构造注入、Approve劫持、私钥提取、MEV利用）将工具接口漏洞的影响从信息泄露和功能滥用升级至不可逆的金融损失。私钥暴露和无限授权的实际发现证实了这些理论向量在真实代码中的存在。

我们呼吁协议设计者将安全性作为协议核心属性而非可选特性。安全控制不应依赖个体工具开发者的主动实现，而应由协议SDK作为不可旁路的默认行为提供。我们发布的静态分析工具和匿名化数据集旨在支持社区持续监测和改进AI agent工具生态的安全状况。
